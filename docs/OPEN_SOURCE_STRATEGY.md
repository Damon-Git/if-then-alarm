# 开源与分发策略

更新日期：2026-06-13。

## 当前决定

《急急如律令》采用自用优先的开源路线，当前不以 Mac App Store 上架或面向普通用户的一键安装为目标。

- 当前不购买 Apple Developer Program 会员。
- 当前不上架 Mac App Store。
- 当前不做 Developer ID 签名、Apple 公证、DMG 或自动更新。
- 保留现有 Tauri `macOSPrivateApi` 透明小窗方案，不为 App Store 单独重做不透明窗口。
- 日常使用继续从本仓库本地构建 `.app`，并按自用回归清单验收。
- 对外分享优先发布源码、构建说明、数据说明和安全边界。

Apple Developer Program 会员费不是公开源码或本地自用的前置条件。只有未来决定上架 Mac App Store，或希望使用 Developer ID 签名和 Apple 公证降低 Gatekeeper 阻拦时，才重新评估购买会员。

## 推荐分发层级

### 第一层：源码仓库

近期默认采用这一层：

- 公开源代码、构建步骤和开发依赖。
- 用户自行安装 Node.js、Rust 和 Tauri 构建环境。
- 用户在自己的机器上运行检查并构建 `.app`。
- 不承诺为未签名产物提供普通消费软件级别的一键安装体验。

### 第二层：未签名测试产物

只有确有分享需要时再考虑：

- 可以在代码托管平台的 Release 中附带未签名 `.app` 压缩包。
- 必须明确标注产物未经 Developer ID 签名和 Apple 公证。
- macOS 可能由 Gatekeeper 阻止首次打开；用户需要自行判断信任来源并通过系统提供的方式确认打开。
- 不建议引导用户关闭 Gatekeeper，也不提供绕过系统安全机制的脚本。
- 发布前仍需执行现有自用检查，并提供构建提交、校验值、数据路径和回滚说明。

### 第三层：签名和公证的直接分发

仅在外部用户数量增加、未签名安装摩擦已经成为实际问题时启动：

- 购买 Apple Developer Program 会员。
- 使用 Developer ID 签名并提交 Apple 公证。
- 评估 DMG、自动更新、崩溃反馈和公开支持渠道。
- 透明小窗可以继续保留，但需要扩大安全与兼容性验收。

### 第四层：Mac App Store

当前不采用。若未来重新考虑，需要先移除或替代 `macOSPrivateApi` 透明 WebView 方案，启用 App Sandbox，并重新验证文件访问、通知、菜单栏、Dock、持久化和数据迁移。该路线视为独立产品阶段，而不是普通打包步骤。

## 开源前必须完成

1. 选择并添加明确的开源许可证。没有许可证的公开仓库不能视为允许他人自由使用、修改和再分发。
2. 检查图片、字体、声音和其他素材的来源及再分发权利。
3. 检查 Git 历史和配置中不存在密钥、账号、真实数据、个人路径备份或其他敏感信息。
4. 补充从干净环境开始的构建说明，以及 Node.js、Rust、Tauri 和 macOS 版本要求。
5. 说明桌面数据位置、备份恢复方式、删除数据方式和未签名应用限制。
6. 为公开 issue 准备最小问题模板，要求提交版本、提交号、macOS 版本和复现步骤，禁止上传真实 `persistence.v1.json`。

## 许可证待决策

许可证是开源前唯一必须由用户确认的近期事项：

- `GPL-3.0`：要求分发的衍生版本继续开放对应源码，适合希望改版仍保持开源的项目。
- `Apache-2.0`：允许更宽松地使用和再分发，同时提供明确专利授权。
- `MIT`：最简短宽松，允许闭源再利用，对衍生版本约束最少。

在许可证确定前，可以继续开发和整理公开准备，但不应把仓库对外描述为已经完成授权的开源项目。

## 不改变的自用边界

- 开源不等于把日常真实桌面 JSON、备份、截图或个人复盘内容提交到仓库。
- 对外贡献不得破坏计时状态机、恢复、历史、复盘、通知和声音时点。
- 未签名公开产物不替代当前长期自用基线；日常 `.app` 是否替换仍按 `docs/SELF_USE_RELEASE_LOG.md` 单独决策。
- 如果未来购买会员或改变分发路线，需要先更新本文和 `docs/DEVELOPMENT_PLAN.md`，再开始发布工程。
