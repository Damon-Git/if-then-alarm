# 交互模型

本文用于冻结《急急如律令》的核心交互语义。它优先于具体实现细节，用来判断后续代码、窗口、菜单栏、小窗和恢复逻辑是否跑偏。

当前最重要的原则是：业务状态、窗口形态、应用可见性、退出恢复必须分层处理，不能互相替代。

## 一、四层模型

### 1. 业务状态

业务状态只回答：本轮进行到哪一步。

页面阶段：

- `setup`：任务创建页，页面标题为“积土成山，积水成渊”。
- `ritual`：仪式台阶段，本轮已经开始但还没有保存复盘。
- `review`：全部香完成，等待复盘保存。

套组状态：

- `idle`：该套尚未开始。
- `burning`：该套正在烧当前炷香。
- `resting`：该套正在休息。
- `completed`：该套全部香完成。

业务状态不应该因为窗口变小、窗口隐藏、菜单栏点击或窗口装饰变化而改变。

### 2. 窗口形态

窗口形态只回答：用户现在以什么形态看应用。

当前定义两个窗口形态：

- 完整窗口：显示主祭台、目标性符箓、预防性符箓、香炉状态、计时面板、历史、设置、复盘等完整操作。
- 小窗：只显示用户创建的 1-3 个 Q 版小香炉；不显示主祭台、符箓、标题、历史、设置、长文本和说明性状态文案。小香炉 hover/focus 可临时显示剩余时间，烧香中有克制烟雾动画。

窗口形态切换不等于业务开始、暂停、完成或放弃。

### 3. 应用可见性

应用可见性只回答：窗口现在是否显示在屏幕上。

当前定义：

- 显示：窗口可见。
- 隐藏：窗口不可见，但应用仍在运行，菜单栏入口仍可重新显示。

隐藏不等于小窗。小窗仍然是可见窗口；隐藏是不可见。

### 4. 退出与恢复

退出与恢复只回答：本轮是否还保留，是否需要用户回来后选择恢复。

当前定义：

- 放弃本轮：清空当前 session，不写历史。
- 保存复盘：写入历史，清空当前 session。
- 恢复本轮：重新载入未完成 session。
- 丢弃恢复：清空未完成 session，不写历史。

关闭窗口不等于放弃，除非用户明确选择“放弃并退出”。

## 二、核心用户动作

### 填写页

点击“开始创造”：

- 业务状态：`setup -> ritual`。
- 窗口形态：进入完整窗口。
- 应用可见性：保持显示。
- 不直接进入小窗。
- 不自动开始任何套组计时。

### 主祭台

点击目标性符箓：

- 如果该套为 `idle`，打开开始确认。
- 确认后，先播放约 2 秒情境符箓燃烧淡出动画。
- 动画结束后，该套进入 `burning`，开始第 1 炷香倒计时。
- 取消后，该套保持 `idle`。

点击“放弃本轮”：

- 打开放弃确认。
- 确认后清空当前 session，返回 `setup`。
- 不写入历史。

点击“保留并收起”：

- 业务状态不变。
- 当前计时继续。
- 窗口形态切换为小窗。
- 应用仍保持显示。
- 计时结束仍应触发应用内提醒或系统通知。

### 小窗

小窗只负责低干扰陪伴和快速回到完整窗口，不承载完整操作。

小窗显示规则：

- 用户创建 1 套，显示 1 个香炉。
- 用户创建 2 套，显示 2 个香炉。
- 用户创建 3 套，显示 3 个香炉。
- 香炉始终单行并排。
- 不显示背景面板、符箓、标题、历史、设置、长文本。
- 小香炉 hover/focus 显示当前剩余时间；移开后不常驻显示倒计时。
- 烧香中的小香炉显示克制烟雾动画；休息中收束或隐藏，完成态不残留持续烟雾。

点击小窗香炉：

- 默认语义：展开完整窗口。
- 不自动开始计时。
- 不自动续香。
- 不自动放弃。
- 不改变业务状态。

悬停小窗香炉：

- 默认语义：查看剩余时间。
- 不展开完整窗口。
- 不自动开始计时。
- 不自动续香。
- 不改变业务状态。

拖动小窗：

- 优先使用香炉横排上方的独立拖拽热区。
- 只移动窗口，不展开完整窗口。
- 拖拽热区不覆盖香炉点击入口。
- 如果从香炉炉身或线香区域按下后明显移动，则复用窗口位置会话拖动小窗，并抑制随后一次点击，避免松手时误展开。
- 小窗拖动后，展开完整窗口再“保留并收起”时，应回到上一次小窗位置；没有有效记录或位置越界时才回退到安全位置。
- 不改变业务状态。

例外说明：

- 未来如果要允许“小窗点击未开始香炉直接开始”，必须重新评估，因为这会让小窗从“陪伴/回到完整窗口”变成“操作入口”。当前阶段不采用该语义。

### 菜单栏入口

点击菜单栏 template icon：

- 只负责显示或隐藏当前窗口。
- 不改变业务状态。
- 不改变完整窗口/小窗形态。
- 不开始、不暂停、不续香、不放弃。

如果隐藏前是完整窗口，再次显示应仍是完整窗口。

如果隐藏前是小窗，再次显示应仍是小窗。

### 左上角关闭按钮

填写页无草稿：

- 可以关闭窗口。

填写页有草稿：

- 弹出确认。
- 用户可以继续填写。
- 用户可以关闭窗口并丢弃草稿。

`ritual` 或 `review` 阶段：

- 必须弹出确认。
- 用户可以关闭弹窗，维持原样。
- 用户可以选择“放弃并退出”：清空当前 session，不写历史，然后关闭程序。
- 用户可以选择“保留并收起”：业务继续，窗口切到小窗。

关闭按钮不应该直接把用户送入小窗，必须经过确认。

## 三、计时与提醒

一次只允许一个套组处于 `burning` 或 `resting`。

专注倒计时结束：

- 如果还有下一炷香，弹出“这一炷香已经烧完。”。
- 用户可以选择休息。
- 用户可以选择立刻续香。

休息倒计时结束：

- 弹出“休息结束，是否继续下一炷香？”。
- 用户确认后进入下一炷香。

最后一炷结束：

- 当前套组进入 `completed`。
- 如果还有其他套组未完成，保留在 `ritual`，等待用户从完整窗口选择下一套。
- 如果全部套组完成且当前是完整窗口，仍停留在 `ritual` 的完成祭台场景，并显示独立收尾入口，文案为“本轮香尽”，主操作为“进入复盘”。
- 如果全部套组完成但当前是小窗，保持小窗完成香炉场景，不直接弹出复盘；小香炉语义提示“点击展开完整窗口查看复盘入口”，用户点击后只展开完整窗口。
- 小窗完成态展开完整窗口不直接进入 `review`；展开失败时保持小窗完成态，用户可以再次点击重试。
- 只有用户在完整窗口点击“进入复盘”后，才进入 `review`。
- 如果最后一炷结束时处于小窗且本轮全部完成，系统通知应提示用户点击香炉回到完整窗口复盘，不直接催促当前小窗完成复盘。

小窗不负责处理复杂决策。遇到休息/续香/复盘等需要决策的场景，应引导用户回到完整窗口。

## 四、复盘与历史

进入 `review`：

- 表示所有套组都已完成。
- 必须来自完整窗口里的“进入复盘”入口，不由最后一炷香结束或小窗展开自动触发。
- 用户需要选择完成/未完成并填写复盘。

保存复盘：

- 写入历史。
- 清空当前 session。
- 返回 `setup`。
- 窗口形态恢复为完整窗口或普通填写窗口，不保持小窗纯香炉形态。

复盘不承担重分析职责。历史分析可以作为独立功能迭代，但不应让复盘流程变重。

## 五、恢复规则

启动时如果存在未完成 session：

- 显示恢复提示。
- 用户选择恢复：恢复业务状态，并进入完整窗口。
- 用户选择丢弃：清空 session，回到 `setup`。

恢复后不直接进入小窗。原因是恢复本轮通常需要用户重新理解当前上下文，完整窗口更合适。

如果恢复时倒计时已经超时：

- 根据保存的计时段推导真实状态。
- 专注已结束则显示烧完提醒或进入完成状态。
- 休息已结束则显示休息结束提醒。
- 最后一炷已结束则进入 `ritual` 的完成祭台场景，等待用户点击“进入复盘”。

## 六、明确不混用的概念

以下概念不能互相替代：

- 小窗不是隐藏。
- 隐藏不是关闭。
- 关闭不是放弃。
- 放弃不是失败复盘。
- 菜单栏入口不是窗口形态切换器。
- 小窗香炉不是主祭台符箓。
- 点击“开始创造”不是开始计时。
- 开始计时必须来自用户对某套意图的明确确认。

## 七、当前建议冻结的交互规则

短期实现应按以下规则收敛：

1. “开始创造”始终进入完整窗口。
2. 完整窗口才显示主祭台和符箓。
3. 目标性符箓是开始计时的主要入口。
4. 小窗只显示香炉，点击香炉只展开完整窗口。
5. “保留并收起”是进入小窗的唯一明确入口。
6. 菜单栏入口只负责显示/隐藏，不切换大窗小窗。
7. 左上角关闭在未保存轮次中必须弹确认。
8. “放弃并退出”才清空本轮。
9. “保存复盘”才写入历史。
10. “恢复本轮”默认回到完整窗口。

## 八、后续待确认问题

这些问题暂不在当前实现中解决，但需要在后续产品设计中明确：

- 小窗透明区域是否需要鼠标穿透。
- 小窗是否需要独立于主窗口的第二个 Tauri window。
- 小窗是否要贴附菜单栏图标，成为真正 popover。
- 小窗里是否允许直接开始未开始套组。
- 休息/续香提醒是否应该自动展开完整窗口。
- 当 Dock 图标隐藏时，完整窗口和小窗的恢复入口是否只依赖菜单栏。
- 未来打包发布是否接受 `macos-private-api` 带来的 Mac App Store 限制。
