# 当前开发计划

更新日期：2026-06-13。

本文是《急急如律令》当前唯一的开发路线图和决策登记表。历史范围、视觉原则和迁移记录继续保留在其他文档中；如果旧文档中的“下一步”与本文冲突，以本文为准。

## 当前基线

当前代码已经具备完整 Web 与 macOS Tauri 自用闭环：

- 创建 1-3 项任务，完成专注、休息、续香、完成祭台和复盘。
- Web `localStorage` 与 Tauri app data JSON 持久化、恢复、备份和损坏保护。
- 完整窗口、小窗、菜单栏入口、系统通知、声音、窗口置顶和 Dock 显示设置。
- 主祭台和小窗正式 v1 素材、线香烟雾、符箓透明 PNG 火焰精灵图及 CSS 回退。
- 主祭台单信息框、小窗 hover 剩余时间、小窗位置会话保持。
- 目标性与预防性符箓 hover/focus 清晰预览，以及 WKWebView 最终尺寸直接渲染。

当前长期自用安装基线仍是 `aa102a2`。2026-06-05 的候选版本已经通过完整验收，但尚未替换日常 `.app`；此后的视觉和 WKWebView 修复也尚未形成新的长期自用候选记录。

## 开发原则

1. 计时状态机、恢复、历史、复盘、通知和声音时点优先于新增体验。
2. 测试和原生冒烟使用隔离数据，不写入或替换真实桌面 JSON。
3. 视觉迭代沿用现有组件、素材 manifest、视觉状态和检查脚本，不新增平行业务状态。
4. 当前阶段不同时推进窗口架构重做、数据结构升级和公开发布。
5. 每个阶段先完成验收和记录，再进入下一个阶段。
6. 项目采用自用优先的开源路线；当前不购买 Apple Developer Program 会员，不为商店审核改变透明小窗架构。

## 第一阶段：形成新的长期自用候选

目标：以当前 HEAD 为起点，确认近期火焰、烟雾、hover 信息框和 WKWebView 清晰度修复可以作为同一候选版本稳定运行。

工作内容：

- 清理路线图中的陈旧待办，使文档与当前实现一致。
- 运行 `git diff --check`、`npm run check:release-self-use`、`npm run check:compact`、`npm run check:tauri-window-roundtrip -- --no-screenshots` 和 `npm run tauri:build`。
- 使用打包 `.app` 走一次 `docs/SELF_USE_REGRESSION_RUNBOOK.md` 核心路径。
- 重点人工检查符箓首次燃烧、普通与减少动态效果、符箓 hover/focus、烧香中单信息框、小窗烟雾、小窗位置保持和完成态恢复。
- 确认复盘保存后历史存在，重开应用后历史仍在，真实桌面 JSON 未被测试污染。
- 无阻断问题后，在 `docs/SELF_USE_RELEASE_LOG.md` 新增候选验收记录。

完成标准：自动检查全部通过，Web 与真实 Tauri WKWebView 无阻断回归，只剩可接受的轻微视觉差异。

## 第二阶段：决定是否升级日常自用基线

目标：把“候选验收通过”和“替换日常使用版本”拆成两个明确动作。

工作内容：

- 验收通过后，由用户决定是否替换当前日常 `.app`。
- 替换前备份 `persistence.v1.json` 和完整备份 JSON，并记录回滚路径。
- 安装后核对版本、Bundle ID、数据版本、历史数量和待恢复轮次。
- 完成一次短核心路径后，将候选记录提升为当前长期自用基线。

本阶段不生成 DMG，不签名，不公证。

## 第三阶段：日常自用体验迭代

目标：只处理真实使用中反复出现的问题，不凭空扩展功能。

建议优先级：

1. 数据安全反馈：让完整备份、恢复结果和损坏文件保留位置更容易理解。
2. 复盘质量：根据真实历史使用情况调整筛选、统计和复盘字段，而不是先增加更多指标。
3. 常用任务体验：只有确认重复书写已经成为明显负担后，再设计受控复用；默认继续保留重新书写动作。
4. 性能和稳定性：观察长时间后台运行、电脑睡眠唤醒、通知和倒计时恢复。
5. 视觉微调：只修复真实 Web/Tauri 截图中暴露的清晰度、层级和遮挡问题。

## 第四阶段：桌面便利能力

本阶段必须逐项获得产品决策后再开发：

- 登录后自动启动。
- 更完整的后台运行和睡眠恢复策略。
- 全局快捷键。
- 小窗位置跨应用重启持久化。

这些能力会扩大平台权限、设置迁移和原生回归范围，不与第三阶段的数据或复盘改动混做。

## 第五阶段：窗口形态评估

当前继续使用单个 Tauri 主窗口，在完整窗口与透明小窗之间切换。只有现有方案出现明确使用障碍时，才评估：

- 透明区域鼠标穿透。
- 独立的第二个小窗。
- 贴附菜单栏图标的 popover。
- 提醒时自动展开完整窗口。

窗口形态变更必须重新验证关闭、隐藏、恢复、Dock、菜单栏、置顶、拖动和当前轮次保护。

## 第六阶段：开源发布准备

当前默认先公开源码，不购买 Apple Developer Program 会员，也不把签名、公证或 Mac App Store 作为开源前置条件。具体边界见 `docs/OPEN_SOURCE_STRATEGY.md`。

- 选择并添加开源许可证。
- 审计代码、Git 历史、素材授权和敏感信息。
- 补齐干净环境构建说明、数据安全说明和贡献约定。
- 需要分享产物时，可单独评估带明确警告的未签名测试包。
- 只有外部用户实际需要顺畅安装时，才重新评估 Developer ID 签名、公证和会员费用。
- Mac App Store 不在当前路线内；若未来重启，必须作为独立阶段重新评估窗口架构和 App Sandbox。

## 决策记录与待决事项

2026-06-13 决策记录：用户同意采用下列建议作为当前默认方案。已经确定的事项不再作为近期开放问题；只有真实使用反馈、平台限制或正式发布目标发生变化时才重新评估。

### 近期必须决策

1. **新候选通过后是否替换日常 `.app`**
   - 已决定：先完成第一阶段验收，验收通过后再单独确认是否替换，不自动覆盖日常 `.app`。

2. **候选验收后的主方向**
   - 已决定：优先进入“日常自用体验迭代”，桌面便利能力继续后排；开源准备以文档、授权和构建可复现性为主，不打断自用稳定性工作。

### 开发前再决策

3. **是否增加历史任务复用**
   - 已决定：暂不增加。只有重复书写被实际证明为负担时，再设计需要用户重新确认的受控复用。

4. **是否登录后自动启动**
   - 已决定：近期不开发；未来若加入，默认关闭并提供显式设置。

5. **是否增加全局快捷键**
   - 已决定：继续后排。开发前必须先明确快捷键用于显示窗口、收起小窗，还是启动任务。

6. **小窗位置是否跨重启保存**
   - 已决定：当前仅保留窗口会话内位置，不做跨重启持久化。

7. **小窗是否继续使用单窗口切换**
   - 已决定：继续使用单窗口切换。独立窗口或 popover 只在现有方案出现明确障碍时评估。

8. **小窗是否允许直接开始未开始任务**
   - 已决定：不允许。继续保持“小窗只观察和返回完整窗口”的低干扰语义。

9. **休息或续香提醒是否自动展开完整窗口**
   - 已决定：不自动展开，继续使用系统通知、声音和用户主动返回，避免抢焦点。

10. **隐藏 Dock 后是否只依赖菜单栏恢复窗口**
    - 已决定：保持菜单栏为主要恢复入口，同时确保重新启动应用也能恢复窗口。

### 开源前决策

11. **开源许可证**
    - 尚未决定：开源前必须在 `GPL-3.0`、`Apache-2.0`、`MIT` 或其他明确许可证中选择。
    - 当前倾向：如果希望衍生分发继续开源，选择 `GPL-3.0`；如果希望更宽松地被复用，选择 `Apache-2.0`。

12. **通知横幅旧占位图是否提前处理**
    - 已决定：继续保持最低优先级；只有未来进入签名公证或更正式的二进制分发阶段时再统一处理。

### 分发路线决策

13. **是否购买 Apple Developer Program 会员**
    - 已决定：当前不购买。公开源码和本地自用不依赖会员资格。
    - 重新评估条件：需要 Developer ID 签名、公证，或重新考虑 Mac App Store。

14. **是否上架 Mac App Store**
    - 已决定：当前不上架，保留现有 `macOSPrivateApi` 透明小窗方案。
    - 重新评估条件：产品目标转为面向普通用户分发，并愿意承担窗口架构、App Sandbox 和商店审核改造。
