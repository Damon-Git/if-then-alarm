## 开发原则

1. 计时状态机、恢复、历史、复盘、通知和声音时点优先于新增体验。
2. 测试和原生冒烟使用隔离数据，不写入或替换真实桌面 JSON。
3. 视觉迭代沿用现有组件、素材 manifest、视觉状态和检查脚本，不新增平行业务状态。
4. 当前阶段不同时推进窗口架构重做、数据结构升级和公开发布。
5. 每个阶段先完成验收和记录，再进入下一个阶段。
6. 项目采用自用优先的开源路线；当前不购买 Apple Developer Program 会员，不为商店审核改变透明小窗架构。

## 第三阶段：日常自用体验迭代

目标：只处理真实使用中反复出现的问题，不凭空扩展功能。

建议优先级：

1. 填写与复盘体验：先处理 2026-06-16 前端审阅发现的表单反馈、复盘主路径和移动端层级问题。
2. 数据安全反馈：让完整备份、恢复结果和损坏文件保留位置更容易理解。
3. 复盘质量：根据真实历史使用情况调整筛选、统计和复盘字段，而不是先增加更多指标。
4. 常用任务体验：只有确认重复书写已经成为明显负担后，再设计受控复用；默认继续保留重新书写动作。
5. 性能和稳定性：观察长时间后台运行、电脑睡眠唤醒、通知和倒计时恢复。
6. 视觉微调：只修复真实 Web/Tauri 截图中暴露的清晰度、层级和遮挡问题。

### 填写与复盘体验整改计划

来源：2026-06-16 对填写页和复盘页做了一轮前端审阅，截图证据保存在 `artifacts/frontend-review/`。结论是当前暖纸色、朱红强调、8px 圆角和符箓/线香语义应继续保留；不引入新的组件库，不做营销页式重绘，只做定向体验修整。

范围：

- 填写页：`src/components/SetupForm.tsx`、`src/components/IntentSetForm.tsx` 和表单相关 CSS。
- 复盘页：`src/components/ReviewPanel.tsx` 和复盘相关 CSS。
- 验收文档：`docs/ACCEPTANCE_CHECKLIST.md`、`docs/SELF_USE_REGRESSION_RUNBOOK.md` 和必要时的 Tauri 桌面验收条目。

整改顺序：

1. **P0 表单错误反馈不误导用户**
   - 用户修改目标性或预防性执行意图后，对应字段的旧错误应立即清除，或在输入过程中重新校验当前字段。
   - 复盘页“一句复盘”为空时，错误必须绑定到 textarea：设置 `aria-invalid`、`aria-describedby`，并复用现有红色边框和浅红底错误样式。
   - 错误信息应靠近对应字段，不能落在补充记录和保存按钮之间造成归属不清。

2. **P1 复盘页突出主任务**
   - 复盘页的主路径是“选择完成度、写一句复盘、保存”，移动端和桌面端都必须先看到这个动作。
   - 本轮摘要、计时模式和任务详情保留，但在移动端应弱化为下方摘要或折叠详情，避免把“一句复盘”推到第四层。
   - 进入复盘页、从恢复弹窗恢复到复盘页、空复盘报错三种路径都要检查。

3. **P1 操作层级调整**
   - “开始创造”继续是填写页唯一主 CTA。
   - “创建任务”应降为次要或幽灵操作，不能在移动端变成比提交更重的横向黑色按钮。
   - 复盘页“放弃本轮”是危险的逃生操作，应低于“保存复盘”的视觉权重；移动端不能抢占标题下方主位置。

4. **P2 空态与微文案**
   - “暂无预防性执行意图”应更像可选空态，而不是系统占位；可以提示“可以不写，遇到容易分心的诱因时再添加”这类轻量解释。
   - 保留现有中文仪式感，但避免让表单字段说明过长或重复。

完成标准：

- `npm run build` 通过。
- 填写页在桌面 `960 × 760` 和移动宽度约 `390px` 下截图无横向滚动、无按钮抢主线、无错误滞留。
- 复盘页在桌面 `960 × 760` 和移动宽度约 `390px` 下，“本次完成度”“一句复盘”和“保存复盘”是主视觉路径。
- 复盘页空提交后，错误样式、错误文案和无障碍属性都指向“一句复盘”字段。
- 不改计时状态机、历史结构、桌面持久化、通知时点、小窗语义或视觉素材 manifest。

## 第四阶段：桌面便利能力

本阶段必须逐项获得产品决策后再开发：

- 登录后自动启动。
- 更完整的后台运行和睡眠恢复策略。

这些能力会扩大平台权限、设置迁移和原生回归范围，不与第三阶段的数据或复盘改动混做。

## 第五阶段：窗口形态评估

当前继续使用单个 Tauri 主窗口，在完整窗口与透明小窗之间切换。只有现有方案出现明确使用障碍时，才评估：

- 透明区域鼠标穿透。
- 独立的第二个小窗。
- 贴附菜单栏图标的 popover。
- 提醒时自动展开完整窗口。

窗口形态变更必须重新验证关闭、隐藏、恢复、Dock、菜单栏、置顶、拖动和当前轮次保护。

## 第六阶段：开源发布准备

当前默认先公开源码，不购买 Apple Developer Program 会员，也不把签名、公证或 Mac App Store 作为开源前置条件。具体边界见 `docs/OPEN_SOURCE_STRATEGY.md`。

- 选择并添加开源许可证。
- 审计代码、Git 历史、素材授权和敏感信息。
- 补齐干净环境构建说明、数据安全说明和贡献约定。
- 需要分享产物时，可单独评估带明确警告的未签名测试包。
- 只有外部用户实际需要顺畅安装时，才重新评估 Developer ID 签名、公证和会员费用。
- Mac App Store 不在当前路线内；若未来重启，必须作为独立阶段重新评估窗口架构和 App Sandbox。
